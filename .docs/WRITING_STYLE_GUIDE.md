# Writing Style Guide

This document captures the writing style, tone, structure, and patterns used in blog posts to maintain consistency across future content.

## Overall Tone

- **Professional but approachable**: Technical depth balanced with accessibility
- **Conversational**: Uses "we", "let's", "I" to create a collaborative learning atmosphere
- **Educational**: Focuses on teaching concepts with practical examples
- **Personal**: Shares personal preferences, experiences, and opinions
- **Balanced**: Acknowledges trade-offs, alternatives, and limitations

## Structure Patterns

### Section Organization
- Use numbered sections (1., 2., 3., etc.) for main topics
- Clear hierarchical headings and subheadings
- Often includes a "Conclusion" or "Conclusions" section at the end
- May include a "Prerequisites" section at the beginning for technical posts

### Paragraph Formatting
- Use `<br />` tags for spacing between paragraphs (not double line breaks)
- Keep paragraphs focused and concise
- Use bullet points for lists of items or features

### Code Examples
- Frequent code blocks with syntax highlighting
- Include language tags (e.g., `typescript`, `bash`, `tsx`)
- Use `title` attribute for code blocks when helpful (e.g., `title="app.ts"`)
- Provide context and explanations before/after code examples
- Show complete, working examples when possible

## Writing Style Characteristics

### Language Use
- **Contractions**: Use contractions naturally ("it's", "we'll", "I've", "don't")
- **Direct address**: Use "you" to speak directly to the reader
- **Questions**: Use rhetorical questions to engage readers and introduce topics
- **Casual interjections**: Include phrases like "Spoiler alert", "TL;DR" when appropriate

### Technical Explanations
- Start with the "why" before diving into "how"
- Use analogies and real-world examples (e.g., e-commerce backend)
- Explain concepts step-by-step
- Reference external resources and official documentation
- Acknowledge when something might seem complex or overkill initially

### Personal Voice
- Share personal preferences explicitly ("I prefer", "I like", "my preferred setup")
- Reference personal experience ("In my experience", "I've found")
- Mention specific projects or tools you use
- Include personal opinions on tools and approaches

### Transparency
- Acknowledge limitations and trade-offs
- Include warnings when appropriate ("Warning:", "It's important to note")
- Mention when something is a matter of preference
- Credit sources and give shout-outs to tools/people

## Content Patterns

### Opening Sections
- Often start with a problem statement or need
- May include a brief overview or TL;DR
- Set context for why the topic matters

### Middle Sections
- Build concepts progressively
- Show practical implementations
- Include multiple examples when helpful
- Reference real-world projects or codebases when relevant

## Formatting Conventions

### Links
- Inline links with descriptive anchor text
- Open external links in new tabs (`target="_blank"`)
- Reference official documentation frequently

### Emphasis
- Use bold for important points or key concepts
- Use code formatting for technical terms, function names, file names
- Use italics sparingly

### Lists
- Bullet points for features, benefits, or items
- Numbered lists for steps or ordered information
- Nested lists when appropriate

### Code Blocks
- Always specify the language
- Use descriptive titles when helpful
- Include comments in code to explain non-obvious parts
- Show complete examples, not just snippets (when possible)

## Technical Writing Patterns

### Explaining Concepts
1. Start with the problem or need
2. Introduce the solution/concept
3. Show basic example
4. Build complexity gradually
5. Show real-world applications

### Code Examples
- Show before/after when demonstrating improvements
- Include comments explaining key parts
- Reference file names and locations
- Show actual usage examples, not just definitions

### Comparisons
- Acknowledge alternatives
- Explain why you prefer one approach over another
- Be fair to other options
- Reference specific tools or libraries

## Common Phrases and Patterns

- "Let's [do something]" - Collaborative tone
- "It's important to note" - Highlighting key information
- "In my opinion" / "I personally" - Personal preference
- "Hopefully" - Gentle, optimistic tone
- "As we will see" - Setting up future content
- "Breakdown:" - Introducing detailed explanation
- "TL;DR:" - Quick summary

## Post-Specific Patterns

### Tutorial Posts
- Clear prerequisites section
- Step-by-step instructions
- Code examples at each step
- Explanation of why each step matters

### Review/Opinion Posts
- Balanced perspective
- Personal experience shared
- Acknowledgment of different viewpoints
- References to specific tools/technologies

### Technical Deep Dives
- Start with motivation/problem
- Build understanding progressively
- Show real-world examples
- Reference production codebases when possible

## Voice and Persona

- **Knowledgeable but humble**: Shares expertise without being condescending
- **Curious**: Shows interest in exploring different approaches
- **Practical**: Focuses on real-world applications
- **Honest**: Acknowledges when things are complex or have trade-offs
- **Encouraging**: Supports readers in their learning journey

## Length and Depth

- Posts vary in length based on topic complexity
- Technical posts tend to be longer with detailed examples
- Opinion/review posts may be shorter and more conversational
- Always aim for completeness in explanations

## References and Citations

- Link to official documentation frequently
- Reference specific blog posts, videos, or resources
- Give credit to tools, libraries, and people
- Include "Read more" sections with additional resources

## Conclusion Style

- Often includes a summary of key points
- May mention future plans or related topics
- Encourages discussion or further exploration
- Ends on a positive, forward-looking note
